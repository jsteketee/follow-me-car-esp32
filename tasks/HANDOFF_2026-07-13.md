## 1. T1 — Serial command RX (per `tasks/T1_serial_command_rx.md`)

- **`src/serial_hal.cpp` / `.h`**: non-blocking RX parser drains `Serial.available()`
  byte-by-byte into a 128 B line buffer (never blocks the loop; no Stream-timeout calls).
  Frames `{"target_speed":..,"target_heading":..}` parse via hand-rolled `strstr`+`strtof`.
  Validation rejects the whole frame on: non-finite values, speed outside
  `[0, rtConfig.maxSpeedMph]`, heading outside `[-360, 720]` (accepted headings normalize
  to `[0, 360)`). Accepted command stored in `CommandData` (module struct, `serial_hal_get()`);
  `lastCmdMs` stamped only on parse+validate success. Resyncs on `'{'`; overflow discards
  to next newline. Non-command lines silently dropped.
- Rate-limited (1/s) `[serial_hal] cmd OK/REJECT (<reason>)` logging added during bench
  debugging — intentionally kept.
- **`include/config.h`**: `CMD_TIMEOUT_MS 300` (compile-time, not rtConfig).
- Host-side unit test of parse/validate/wrap logic passed (12/12 checks, incl. NaN/range
  rejection, `-90→270` normalization, seam wrap `yaw=10,target=350 → +20`).

## 2. Mode / control changes (user-directed, supersedes parts of T1 brief + PROJECT_PLAN)

- **`NavMode::REMOTE` added** (`nav.h`, `nav.cpp` mode string, `control.cpp`,
  `dashboard.cpp` /mode handler + button + navState, `oled.cpp` mode label).
- **`DEFAULT_NAV_MODE = REMOTE`** (was FOLLOW_ME) — car boots Pi-commandable.
- **Boot heading hold**: `control_init()` seeds `_remoteHeadingDeg` from boot yaw
  (imu_init blocks until first rotation vector, same guarantee fusion_init uses). In
  REMOTE, steering PID holds that heading immediately at boot, zero throttle.
- **Failsafe semantics changed from spec**: on cmd timeout (>300 ms), throttle-only
  failsafe — throttle neutral + throttle PID reset, but **steering stays active holding
  the last commanded (or boot) heading**. Rationale: centering wheels mid-turn on a comms
  blip makes the car plow straight. Car stays in REMOTE; throttle auto-resumes on next
  valid frame. Each valid frame overwrites the held heading.
- Heading error wrapped to [-180, 180] (short way), same while-loop pattern as
  `fusion.cpp correct_bearing()`; mirrors FOLLOW_ME's structure so tuned gains transfer.
- **FOLLOW_ME control block commented out** in `control.cpp` (mode now behaves as
  STOPPED; Pi owns follow logic). Enum/dashboard button remain. TEST / THROTTLE_TEST /
  STOPPED unchanged. Serial frames never change nav mode; dashboard `/mode` (incl.
  STOPPED kill switch, which still centers steering) always wins.

## 3. WiFi rework (`wifi_config.cpp/.h`, `config.h`, `include/secrets.h`)

- **STA-first, non-blocking**: boot tries "cupcakes", then hotspot "jackspot" (creds in
  gitignored `secrets.h`; `WIFI_HOTSPOT_SSID/PASS` added), then falls back to soft-AP
  "Follow Me". 7 s per network (`WIFI_STA_TIMEOUT_MS`). State machine advances in
  `wifi_update()`; `wifi_init()` returns immediately so boot/control aren't delayed.
  mDNS/OTA/telnet start once a network is up. Auto-reconnect on STA drop (no runtime AP
  fallback after initial connect).
- **Hostname renamed** `"Follow Me"` → `followme` (spaces are invalid for mDNS/DHCP):
  dashboard `http://followme.local/`, telnet `nc followme.local 23`, OTA
  `--upload-port followme.local`. AP-fallback dashboard remains `192.168.4.1`.
- New `WifiInfo` state struct via `wifi_get()` (online, sta/AP, ssid, ip, onlineSinceMs).
